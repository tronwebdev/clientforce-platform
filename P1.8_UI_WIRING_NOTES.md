| **Logs** | `tabLogs` | ‚úÖ | the **events timeline** ‚Äî the P1.7 event stream for the campaign (sent/delivered/open/click/reply/branch/stage) **+ send-boundary refusals/adjustments** (typed 422 refusal reasons; faux-"Re:" strip+audit events) as visually distinct amber rows ‚Äî the audit trail is user-visible, never just a server log |
| **Settings** | `tabSettings` | ‚úÖ | Channels & senders + **Tracking & compliance** toggles (`settingsToggles`) ‚Äî unsubscribe/opt-out lives here |
| Calls | `tabCalls` | ‚¨ú defer | voice ‚Äî later phase |
| Stats | `tabStats` | ‚¨ú defer (or basic) | analytics |

**The two highest-value surfaces for "the agent is aware of every activity":**
1. **Lead detail drawer** (`leadDrawerOpen`, fixed right drawer, `rgba(12,20,15,.45)` scrim) ‚Äî per-lead
   **activity timeline**: every captured event (sent, opened, clicked, replied+intent, stage change).
   This is the human-visible proof of the engagement-awareness loop. Wire it to the lead's `Event` rows.
2. **Logs tab** ‚Äî the campaign-wide event feed. Same data, campaign scope.

**Inbox = the loop made visible:** a reply lands ‚Üí P1.7 classifies intent ‚Üí it appears under the matching
`inboxCats` category with the lead's thread. The category chips ARE the intent labels. Mark-done +
row-menu actions map to lead/enrollment mutations.

## F. Three-tier sender model ‚Äî UI gap check (Settings sender detail drawer)

**Finding: the sender detail drawer already renders all three tiers ‚Äî almost no display gap.** It's a
conditional view (`Settings.dc.html`) that shows the right blocks per `SenderConnection.type`:

| Drawer block | Flag | Covers |
|---|---|---|
| Dedicated IP (IP address mono, **blacklist monitoring**, **ISP reputation** bars) | `sd.isDedicatedIp` | Tier 2 (dedicated) |
| **Warm-up tracker** (Day / current limit / target / progress bar) | `sd.hasWarmup` | warmup ramp ‚Äî *exists* in proto |
| OAuth2 auth (status, token expiry, permissions granted) | `sd.hasAuth` | Tier 3 Gmail/Outlook |
| SMTP + IMAP field tables | `sd.hasSmtp` | Tier 3 custom SMTP |
| Domain authentication (SPF/DKIM, **Re-check DNS**) | always | all tiers |
| Health/reputation score + sent-today + assigned agents | always | all tiers |

> Correction to an earlier note: **warmup UI does exist** (`sd.hasWarmup` block). It's just not wired in
> P1 ‚Äî leave it visible, drive it later. The dedicated-IP, OAuth, and SMTP detail views are all designed.

**The ONE genuine UI gap ‚Üí the upgrade *action*.** The drawer *displays* a dedicated IP once assigned
(`isDedicatedIp === true`), but there is **no CTA/flow for a shared-tier user to upgrade to a dedicated
IP** (request/purchase ‚Üí provision subuser+IP ‚Üí warmup starts). When we build Tier 2, add:
- An **"Upgrade to dedicated IP"** action on a `CF_MANAGED` shared sender (button in the drawer or
  sender row), gated by a **volume threshold + plan/price** (a low-volume dedicated IP *hurts*
  deliverability ‚Äî gate it).
- A short **confirm/checkout** step (it's a paid upgrade), then the drawer flips to the existing
  `isDedicatedIp` view automatically once provisioned.
This is **later-phase** (Tier 2), not P1 ‚Äî flagged here so it's not forgotten. Everything else for the
three tiers already has a designed surface.
