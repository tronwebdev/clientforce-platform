(flex, `padding:28px 32px`), with a `‹ Back` (white) and a gradient `Next` button pinned bottom of the

| Step | Anchor | Wire in P1.8? | Backed by |
|---|---|---|---|
| 1 · **Setup Agent** | `isSetup` | ✅ | identity + **goal** (6 cards, 3-col grid `repeat(3,1fr)`, gap 9) → reveals **Knowledge base** list (add URL/doc) → "Let AI orchestrate" vs "from scratch" channel picker | agent create + P1.2 ingest |
| 2 · **Design Sequence** | `isSequence` | ✅ (read-only ok) | step cards: `Step N` label + channel chip + `✦ AI draft` badge + `✎ Edit`; **step editor modal** (`editorOpen`) + **delay modal** | the **planned CampaignGraph** (P1.4) |

| 4 · **Enable Lead Capture** | `isCapture` | ⬜ defer | inbound capture toggles | later phase |
| 5 · **Channels & senders** | (last) | ✅ | readiness banner + **email senders** list + **volume/deliverability limits** | P1.5 sender config |

**Email-slice essentials in this wizard:**
- **Goal cards** (`goals`, `g.onPick`) — the picked goal feeds the planner (P1.4). Show the confirmed
  goal as the green pill divider `✓ Goal: {{ goalTitle }}` exactly as the prototype does.
- **Knowledge base** (`knowledgeSources`, add-URL row `isAddUrl`) — each added URL kicks off P1.2
  ingestion; show `IngestStatus` (PENDING→INGESTING→READY/FAILED) on the row.
- **Sequence step cards** render the **planner's graph** (P1.4): one card per `step` node, the channel
  chip = email, the delay between = the `delay` node, the reply-branch = the `branch` node. `✦ AI draft`
  badge stays (the copy *is* AI-drafted).
- **Email senders + volume limits** (step 5) = the P1.5 sender + daily-cap/window config.
- Launch → the dark "**Your agent is live**" success state (`#0C140F` bg, `fadeUp` animation) — keep it.
- **Citations (P1.3 → UI contract).** The distiller returns per-field
  `citations[] = { chunkId, sourceId, sourceLabel, sourceType, locator, quote }` (quote = verbatim
  passage; locator = page/Q#/URL path). Three surfaces render them — port from the updated prototypes:
  1. **About-your-business card footer** ("Grounded in" chips → quote reveal + fields backed),
  2. **gap checker "✓ Found in your docs"** — covered fields are chips → same quote reveal (evidence,
     not assertion),
  3. **Settings → Brand kit → Agent summary** — per-section provenance chip (`N sources` / `✦
     AI-inferred` / `✎ edited`) + expanded "Grounded in" chips + quote reveal; user edits flip the
     section to `✎ edited — overrides docs` with ↺ Revert (edits persist as context overrides, per
     the existing gap-resolution behavior); AI-delegated/inferred fields show the amber ✦ note.
  Chunk ids never surface raw in UI — always source label + locator.

## B. Campaign View — tabs (`Campaign View.dc.html`)

Tab row drives `tab===` flags. Wire this subset for the email slice:

| Tab | Anchor | Wire? | Shows |
|---|---|---|---|
| **Steps** | `tabSteps` | ✅ | the running sequence (`Main sequence · N steps`, `isSeqView`) — render the persisted CampaignGraph + per-step counts |

| **Logs** | `tabLogs` | ✅ | the **events timeline** — the P1.7 event stream for the campaign (sent/delivered/open/click/reply/branch/stage) |
| **Settings** | `tabSettings` | ✅ | Channels & senders + **Tracking & compliance** toggles (`settingsToggles`) — unsubscribe/opt-out lives here |
| Calls | `tabCalls` | ⬜ defer | voice — later phase |
| Stats | `tabStats` | ⬜ defer (or basic) | analytics |

**The two highest-value surfaces for "the agent is aware of every activity":**
1. **Lead detail drawer** (`leadDrawerOpen`, fixed right drawer, `rgba(12,20,15,.45)` scrim) — per-lead
   **activity timeline**: every captured event (sent, opened, clicked, replied+intent, stage change).
   This is the human-visible proof of the engagement-awareness loop. Wire it to the lead's `Event` rows.
2. **Logs tab** — the campaign-wide event feed. Same data, campaign scope.

**Inbox = the loop made visible:** a reply lands → P1.7 classifies intent → it appears under the matching
`inboxCats` category with the lead's thread. The category chips ARE the intent labels. Mark-done +
row-menu actions map to lead/enrollment mutations.

wired** this phase. Phase-1 deliverability = the **per-sender daily limit** (wire it) + the **suppression

a **later phase** — leave the warmup block visible but inert; the sender model reserves a `warmupState`
field so driving it later needs no refactor.

## F. Three-tier sender model — UI gap check (Settings sender detail drawer)

**Finding: the sender detail drawer already renders all three tiers — almost no display gap.** It's a
conditional view (`Settings.dc.html`) that shows the right blocks per `SenderConnection.type`:

| Drawer block | Flag | Covers |
|---|---|---|
| Dedicated IP (IP address mono, **blacklist monitoring**, **ISP reputation** bars) | `sd.isDedicatedIp` | Tier 2 (dedicated) |
| **Warm-up tracker** (Day / current limit / target / progress bar) | `sd.hasWarmup` | warmup ramp — *exists* in proto |
| OAuth2 auth (status, token expiry, permissions granted) | `sd.hasAuth` | Tier 3 Gmail/Outlook |
| SMTP + IMAP field tables | `sd.hasSmtp` | Tier 3 custom SMTP |
| Domain authentication (SPF/DKIM, **Re-check DNS**) | always | all tiers |
| Health/reputation score + sent-today + assigned agents | always | all tiers |

> Correction to an earlier note: **warmup UI does exist** (`sd.hasWarmup` block). It's just not wired in
> P1 — leave it visible, drive it later. The dedicated-IP, OAuth, and SMTP detail views are all designed.

**The ONE genuine UI gap → the upgrade *action*.** The drawer *displays* a dedicated IP once assigned
(`isDedicatedIp === true`), but there is **no CTA/flow for a shared-tier user to upgrade to a dedicated
IP** (request/purchase → provision subuser+IP → warmup starts). When we build Tier 2, add:
- An **"Upgrade to dedicated IP"** action on a `CF_MANAGED` shared sender (button in the drawer or
  sender row), gated by a **volume threshold + plan/price** (a low-volume dedicated IP *hurts*
  deliverability — gate it).
- A short **confirm/checkout** step (it's a paid upgrade), then the drawer flips to the existing
  `isDedicatedIp` view automatically once provisioned.
This is **later-phase** (Tier 2), not P1 — flagged here so it's not forgotten. Everything else for the
three tiers already has a designed surface.
