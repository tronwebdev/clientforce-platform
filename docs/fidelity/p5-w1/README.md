## 1 · Score fixture matrix (deterministic formula, DEC-083)

`score = Σ wᵢ·componentᵢ / Σ wᵢ` — weights bounce .40 · spam .35 · delivery
.15 · reply .10 (delivery excluded + renormalized when no delivery signal
exists in the 7-day window). Component zero-points anchor on **2× the DEC-019
breach levels**: bounce 0 pts at ≥6% (breach 3%), spam 0 pts at ≥0.2% (breach
0.1%); delivery 100 pts at ≥98% / 0 at ≤85%; reply 100 pts at ≥2%.
Sample floor = F1's `SIGNAL_MIN_SENDS` (none <20 · low 20–49 · ok ≥50):
below 20 sends there is **no score and no gate** — "warming / low data".
Auto-pause below **40**, recovery at **≥55** (hysteresis band 40–54 keeps the
prior state).

| fixture | sent | dlv | bounce | spam | reply | → score | state | floor |
|---|---|---|---|---|---|---|---|---|
| 19 sent, all bounced (below floor) | 19 | 0 | 19 | 0 | 0 | **—** | low_data | none |
| clean 100, no delivery signal | 100 | 0 | 0 | 0 | 0 | **88** | healthy | ok |
| fully engaged (98% dlv, 2% reply) | 100 | 98 | 0 | 0 | 2 | **100** | healthy | ok |
| DEC-019 bounce breach 3% | 200 | 190 | 6 | 0 | 4 | **77** | healthy | ok |
| 2× both breaches + weak delivery | 1000 | 850 | 60 | 2 | 5 | **3** | unhealthy | ok |
| bounce storm 25/25 | 25 | 0 | 25 | 0 | 0 | **35** | unhealthy | low |
| hysteresis band sample | 1000 | 900 | 20 | 1 | 0 | **50** | healthy* | ok |
| recovery-grade clean window | 80 | 78 | 0 | 0 | 2 | **99** | healthy | ok |

\* 50 sits in the 40–54 band: it stays `unhealthy` when the prior state was
unhealthy, `healthy` otherwise — pinned both ways in `health.test.ts`.

## 2 · Ramp curve walk (warmup scheduler, curve v1)

Geometric ramp: **2% of the configured daily limit on day 1 → 100% on day 45**
(the canon length — the Settings prototype's "Day N of 45"), absolute floor
10/day, always clamped to the configured limit. Effective boundary cap =
**min(warmup cap, dailyLimit)** — both checks run in the existing cap rail
(`DAILY_CAP_REACHED`, no new enum member for caps). Warmup is **triggered,
never retroactive** (DEC-019): it rides `warmupState.startedAt`, stamped by
the create endpoint on NEW senders — every pre-W1 sender keeps byte-identical
send behavior (regression-pinned).

Boundary proof (integration, real PG): a **day-3** sender (target 200, cap 10)
with 10 sends today refuses `DAILY_CAP_REACHED — warmup cap 10 (day 3 of 45)`;
a second sender the same day at 9 sends **sends** (the count is per-SENDER, on
the new indexed `Message.senderId` column). A day-45 sender with `dailyLimit 2`
refuses on the untouched legacy `sender limit 2` check — min() holds from both
sides. SMS twin pinned (day-2 cap on `channel:"sms"` counts).

## 3 · Auto-pause refusal + recovery walk

End-to-end at the API (e2e, real PG + RLS, capturing transport):

1. Sender with 30 sends in-window; SendGrid webhook delivers **9 bounces +
   1 spamreport** → the webhook fast path recomputes: bounce 30%, spam 3.3% →
   `healthState.state = "unhealthy"` persisted, **one**
   `sender.health_collapsed.v1` Event row (guarded persist — a second
   recompute emits nothing).
2. Next `POST /senders/test-send` through the FULL boundary → **422
   `SENDER_UNHEALTHY`** (typed; enrollment-path disposition stays PAUSED =
   reversible, the TENANT_SUSPENDED pattern — no send-path fork).
3. Recovery: a healthy snapshot (what the sweep writes after a clean window)
   → the same sender **sends again**; the integration suite additionally pins
   unhealthy → healthy emitting one `sender.health_recovered.v1`, and the
   window-drain path (unhealthy → `low_data`, gate open, `lowData: true` on
   the recovery event).

Rail order pinned (the "make N rails fail, assert the earliest wins" pattern):
suspension ▸ `SENDER_DISABLED` ▸ **`SENDER_UNHEALTHY`** ▸ window/caps ▸
opt-out/suppression — an unhealthy sender with a suppressed contact refuses
`SENDER_UNHEALTHY`; a PAUSED unhealthy sender refuses `SENDER_DISABLED`.

## 4 · DNS job states (SPF/DKIM/DMARC)

Real lookups (SendGrid domain-auth API for CF_MANAGED SPF/DKIM + direct
`_dmarc.<root>` TXT), per-record `{ status, pass, detail, expected?, found?,
lastCheckedAt }` persisted into `domainAuthStatus` (the shape the Settings UI
already reads, extended additively). States walked in tests:

| state | when | example detail |
|---|---|---|
| `verified` | the check that wrote `lastCheckedAt` confirmed the record | "Policy record present at _dmarc.clientforce.io" |
| `failed` | provider/DNS answered and the record is wrong/missing — with a **copyable expected record** | "Not configured — no TXT record at _dmarc…" + `expected` |
| `unchecked` | the check could not run (no provider key, API 5xx, resolver timeout) | "Lookup failed (ETIMEOUT) — could not verify" |

**Never cached-as-verified** (pinned e2e): a verified sender whose next
re-check hits a resolver timeout is REPLACED with `unchecked` — a stale pass
is never served. SMS senders return null / `POST :id/dns-check` → 400 (no DNS
posture — honest, not a fake pass). Every check REPLACES the whole record set.
